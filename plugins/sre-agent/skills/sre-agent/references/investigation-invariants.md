# RCA Honesty Floor

This is the shared honesty floor; other files point here instead of restating it.

- Claims cite observations. Material claims without a cited observation are named as gaps, not conclusions.
- The incident report is a hint, not truth — but read it, thread included. The incident's own record AND its discussion thread (human comments, transfers, owner notes, prior RCA/mitigation, linked change/rollout notes) are required intake orientation evidence: read and summarize them before declaring anything blocked-unreachable or producing an RCA, since they often carry the human-known cause or mitigation. Reported RootCause, Mitigation, and timeline narrative still stay claims until corroborated by a different, non-narrative source (telemetry, code, config, or control-plane state). The incident's own body/description is itself a first-class intake source: retrieve and parse it — not only the discussion-thread events — before declaring incident-provided keys absent or asking the reporter to supply them; a field missing from a partial retrieval or projection is a retrieval gap, not absent data (apply "Empty is not absent" to intake).
- Symptom is not cause. A theory names the cause and the mechanism linking it to the symptoms, not just a failure signal.
- A1 rca_target: the RCA target is the underlying measured/impacted failure the signal evidences, not why the alert fired; the alert is evidence, not the root-cause object.
- A cause is not root until its mechanism is verified against an authoritative source for that mechanism (implementation, configuration, runtime/control-plane state, or operational docs as fits the claim) — not inferred from symptom telemetry alone.
- **Bind to the authoritative failing population.** An RCA must explain the failures the signal measures — the actual failed units or a justified keyed cohort — from the authoritative source that drives the alerted metric (not an adjacent system log; reach it via the access invariant's cross-source pivot), followed by their identity/lineage keys to the producer. A nearby error, coincident count, or empty join may *rank* leads, but may **promote or refute** a cause only when bound to that population; if that population's source is honestly unreachable, that is a blocked/proximate disposition with a receipt, not a rooted verdict.
  - Proximate is not root: a cause that only restates the failure is an open lead, not the verdict, while an answerable upstream 'why' remains.
  - Empty is not absent: a negative or zero-row result refutes or blocks only from a probe fit for that population — authoritative signal, covered scope, compatible join keys — **equally when refuting a rival**; an unfit empty (guessed signal, wrong scope, incompatible/unjoined IDs) is a probe defect, not evidence. Re-probe within the follow-up budget (see [grading-rubric.md](grading-rubric.md)) before recording absence.
  - Aggregate is not mechanism: a rate, count, ratio, or threshold breach shows that something failed, not why; bind the mechanism to the enumerated population above before asserting OR excluding it.
- Dead-end honestly — a blocked verdict is a delegation, not a dead-end: once the honesty, access, and probe-fitness gates establish that the decisive evidence is genuinely unreachable because it needs a human-only or out-of-band capability (elevation/JIT, VPN, a GUI-only explorer, a monitor-rule export, an owner-only state reader), do not stop at the proximate cause — name the engineer's suggested next step and hand off an operator-executable manual investigation kit so the on-call engineer can finish (canonical shape in `artifact-contracts.md` §`6_report/`).
- Access is confirmed, not assumed-blocked: a failed or misused probe is non-diagnostic; only a terminal missing-command or denied-after-valid-auth error — after a correct, authed, non-guessed invocation — proves a capability unavailable. One bad probe never globally disables an evidence source. A documentation gap is not an access failure: a signal, table, or source named-but-not-confirmed (or missing) in service knowledge is an unattempted probe, not a block — when a working read-only query capability is in hand, attempt the canonical probe against the service's own evidence source, resolving a docs-versus-live naming difference via schema/catalog discovery, before recording it unavailable. The system that surfaced the symptom (incident-management, monitoring, or alerting) is not necessarily the one that answers the upstream "why": the next causal layer usually needs the service's own runtime evidence — often a different source, sometimes a different signal or namespace within the same source — reached by following the failing units' correlation/identity keys. Like any unattempted source above, it stays open-answerable, not a block, until a fit-for-purpose probe of it is spent or no such source exists.
- "Likely" has a floor: use the graded verdict and report wording from [grading-rubric.md](grading-rubric.md).
- Gaps stay visible. No clean all-clear while a material gap, open question, or partial result remains.
- Recurrence informs, never overrides. An incident is rarely unique in time: check whether a matching or overlapping incident identity — signal/error signature, affected operation/component, affected entity/resource/cohort, scope boundary, or owning service/team — has fired before or concurrently, and let prior occurrences shape which hypotheses lead and which mechanisms/mitigations to test first. Prior incident records, verdicts, and mitigations are claims, not current truth: a prior benign/noise verdict never licenses a clean all-clear while live evidence shows real failures, and a prior root cause or mitigation is re-verified against this incident's evidence before it can close, confirm, or downgrade a lead. Recurrence may classify incidents sharing the recurrence identity into a canonical/duplicate relationship; it still never imports another incident's cause, verdict, or mitigation as truth. This extends to cross-signal cascade candidates: a concurrent incident on a shared resource or an upstream dependency edge (e.g. a provider incident preceding this one) is a hypothesis that chooses what to check next, never proof of this incident's cause — a downstream attribution requires this incident's OWN evidence, not the related incident alone (the attribution-tier bar is in [grading-rubric.md](grading-rubric.md)).
- Roles stay honest: scout stays neutral with surfaces, hypotheses, and questions; grader stays skeptical; report wording never exceeds the grader's verdict.
- Collectors are read-only for incident and host systems, never echo secrets or tokens, and each subagent writes only its own stage. The one sanctioned mutation of the incident system is the Poster's confirm-bound, live update with a capability-owned idempotency/audit marker when the run authorizes it (explicit brief authorization AND a non-gated posting capability; see `subagents/poster.md`); collectors and all other roles remain strictly read-only.
- Incident-facing posts are self-contained: their body, footer, and marker never expose locators of the agent's private execution state, such as run id/root, `.investigations/`, stage/artifact paths, tool run paths, or local absolute workspace paths. External references must be owner-resolvable — e.g. the incident id, a telemetry query/dashboard, a service-scoped source/config path, an accessible knowledge link, a work-item/build/correlation id, or an owner/team — keyed on resolvability, not this list. Internal locators stay only in internal artifacts.
