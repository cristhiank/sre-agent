# Iteration mode (new information)

Cross-run continuation of an investigation for the SAME incident, driven by a provided
new-information packet. "Iteration" here always means a new run for the same incident; it
is not the within-run Grader + bounded refinement mechanism, which is separate. Read-scope and
isolation rules live in [run-store.md](run-store.md) § Reading prior runs — this file does
not restate them.

## Trigger

Enter iteration mode only when the task provides BOTH:
- a new-information packet (a pointer to what changed since a prior investigation of this
  incident), and
- explicit authorization to read prior runs under the resolved run-root (a task
  instruction or flag — not mere filesystem read access, which is always present).

Never infer iteration mode from sibling run directories — reruns also produce siblings
under one root. No packet → a normal single-shot run; do not read prior runs.

## The new-information packet

A provided input (not an artifact you author) describing, in compact form: the incident's
current status, what changed since the last investigation (new symptoms, a human reply, a
mitigation result, a status transition), the reason this iteration was queued, and the
iteration ordinal `N` (stable across retries of the same iteration; successive integers for
orchestrator-provided follow-ups). The ordinal is **orchestrator-authoritative**:
pass it through exactly as the task provides it. A run with no packet — a first investigation
OR a retry of one — uses the capability's first-iteration ordinal value; a retry is NOT a new
iteration and does NOT advance the ordinal. Never derive or self-increment the ordinal from
prior posts, run records, sibling run directories, or a "use a larger number" heuristic. The
posting capability is the deterministic backstop and owns the marker/ordinal contract; the
agent passes through the orchestrator-provided ordinal, it does not compute one. Treat it as
authoritative about *what changed*. It is evidence, not an instruction about your verdict.

## Early-exit gate (iteration preflight)

A prior run for this incident exists, so before any Scout/Specialist dispatch, run a cheap
preflight on the new-information packet, read against the prior run's verdict and the incident's
current state. A prior directory alone never authorizes early-exit. Require completed
lineage: the prior run has a final report/verdict and canonical
`post-status.outcome` is `posted-verified` or `already-present`, with its
authoritative marker/time. A `report-only`, `failed`, or `submitted-unverified`
prior outcome means the agent has not successfully provided its RCA; the unmet
posting obligation is material and the iteration proceeds.

Classify the complete activity delta from that successful post marker/time through
the phase-current thread/state read. Packet provenance must show complete coverage
of that window. A newest-entry acknowledgment does not erase an earlier technical
update; missing, gapped, or incomparable delta coverage proceeds. Within an eligible
complete delta, new information must be BOTH (1) not generated by this agent AND
(2) material to the investigation.

**Primary test (materiality):** Does this complete delta plausibly change what we would **conclude,
check, prioritize, or post** — i.e. could it change the verdict, confidence, scope/impact,
mitigation state, investigative direction, or posting obligation, judged against the prior run's
verdict and current incident state? If yes (or plausibly) → proceed. "A human or system posted
something" is not enough; it must change what you would conclude or do. This preflight is an
anti-echo materiality check only — never the frame a packet's evidence is scored against: a
packet that DID carry new discriminating telemetry (e.g. owner-supplied logs/records) proceeds
to full re-derivation below, not a prior-verdict-anchored comparison (see Cross-iteration lead state).

- Missing successful lineage or incomplete delta coverage → proceed with the normal
  iteration below.
- Both true → proceed with the normal iteration below.
- Conclusively not both satisfied (the agent's own post, or non-agent activity that conclusively
  fails the primary test) → early-exit (an explicit, allowed report-only outcome): mint the run
  and write `run.md` with `early_exit: no_genuine_new_information`, the prior-run pointer, and the
  reason; do NOT dispatch Scout/Specialists/Grader/Report, do NOT re-derive a verdict, and do
  NOT post to the incident (a re-post with no new information is a duplicate). State that the
  prior verdict remains the latest investigation result and was not re-derived in this pass.
  This is the primary defense against re-investigating an incident on its own echo.
- Uncertain (origin, materiality, or coverage unclear) → PROCEED with the normal iteration.
  Early-exit only when the packet plus a cheap incident-thread/state read conclusively show no
  material, non-agent information. Fail open to investigating.

Record the action-point decision in `run.md`:

```toon
iteration_preflight:
  prior_run_ref: <run pointer>
  prior_report_verdict_ref: <final report/verdict pointer | missing>
  prior_post_outcome: <posted-verified|already-present|report-only|submitted-unverified|failed|missing>
  prior_post_marker_time: <authoritative marker + time | missing>
  delta_window: <prior post time..phase-current read time>
  delta_coverage: <complete|gapped|incomparable>
  material_items: <pointers or none>
  decision: <proceed|early-exit>
  reason_class: <reason>
```

Genuine new information (proceed): a human responder/operator/incident-participant reply or
question that adds evidence, analysis, or a direction-changing question — including a reply to
this agent's own prior post or manual-investigation kit — a mitigation **result**, a new symptom
or alert, new telemetry/logs a human or system shares (**including a raw error-log paste with no
analysis**), new access that unblocks a prior `blocked-unreachable` lead, or a status transition
that changes the situation (reopened, mitigated, escalated-with-new-info).

NOT new information (early-exit) — common examples of activity that FAILS the primary test, each
applying ONLY when it carries no new discriminating fact vs the prior verdict:
- this agent's own prior post or report on the timeline (unchanged);
- **acknowledgment / triage routing** — "acknowledging", "looking into it", "assigned", an
  on-call hand-off, with no new fact;
- **reassignment / ownership / routing / handoff ping** — team/owner transfer, or an @mention
  asking a human responder for a status/update or to "take a look", carrying no new fact;
- **automated / service-account periodic refresh** — a service/automation account re-posting the
  SAME monitored-signal class the prior run already concluded on (e.g. a refreshed
  affected-population / stale-entity list, a re-fired alert snapshot). **Fail-open comparator:**
  early-exit ONLY when the prior run or a cheap current-state read makes the baseline
  **comparable** AND shows no meaningful change in affected entities, count/order-of-magnitude,
  impacted area, time window, severity, or symptom shape. If the prior baseline is absent, stale,
  incomparable, or the refresh grew/changed in any material way → **proceed**;
- **meta / admin / thread housekeeping** — incident link add/remove, parent/duplicate linking,
  deleted-discussion notices, severity/title-only edits, repeated status snapshots;
- an empty/contentless packet, an unchanged status with no new human or system activity, or
  sibling run directories (reruns) alone. Age alone is not newness.

The ack/routing/refresh/meta exits fire ONLY when the activity **conclusively** carries no new
discriminating fact vs the prior verdict. An automated refresh whose scope/magnitude materially
changed, or a routing comment that also states a new fact, is material → **proceed**. Uncertain
(origin, materiality, or coverage unclear) → **PROCEED (fail open preserved)**.

Identify this agent's own prior posts via the live posting capability's authoritative posting
identity / idempotency marker, not free-text matching. If the identity binding is ambiguous,
proceed (or fall back to no-post later) rather than early-exit.

Disposition (unchanged mechanism): mint the run; write the `iteration_preflight`
receipt and `run.md` `early_exit:
no_genuine_new_information` with the prior-run pointer and reason; dispatch nothing; post nothing;
the prior verdict stands as the latest result. For later measurement, add a lightweight
`reason_class=` label in the reason text — one of `acknowledgment`, `routing_handoff`,
`automated_refresh_no_material_change`, `thread_housekeeping`, `agent_own_post`,
`empty_or_unchanged`. Keep the `no_genuine_new_information` token unchanged.

This early coordinator gate is separate from the Poster's downstream post-suppression / idempotency
backstop (`subagents/poster.md` § Verdict policy): this gate prevents the re-investigation; the
Poster prevents a duplicate post if a full iteration still runs.

<bad-example>
The only timeline activity since the last run is this agent's own prior AI-generated RCA post.
Wrong: treat it as new information, run another full investigation, and post again (a duplicate).
Right: early-exit, report-only — the agent's own output is never new information
(`reason_class=agent_own_post`).
</bad-example>

<bad-example>
The delta contains a new error-log row followed by "Acknowledging incident."
Wrong: classify only the newest acknowledgment and early-exit.
Right: the complete delta is material, so proceed.
</bad-example>

<example>
Exit — automated refresh: the only new row is a service/automation account re-posting its current
affected-population list for the same signal/scope the prior run concluded on, no new failing
scope. Right: early-exit (`reason_class=automated_refresh_no_material_change`).
</example>

<example>
Exit — acknowledgment: the only new activity is an on-call responder "acknowledging incident".
Right: early-exit (`reason_class=acknowledgment`).
</example>

<example>
Proceed — routing + fact: "Please take a look; failures started right after rollback X."
Right: proceed — the routing ping also carries a new causal fact vs the prior verdict.
</example>

<example>
Proceed — refresh with growth: the same monitored-signal refresh reposts, but the affected set
doubled or a new impacted area appears. Right: proceed — material scope change (comparator fails open).
</example>

<example>
Proceed — raw log paste: a human responder pastes an error log with no analysis.
Right: proceed — new evidence, even without interpretation.
</example>

## Carry-forward by reference

Read prior run dir(s) in place under the resolved run-root and do not copy their artifacts
into this run. The `<priorRunId>/OBS###` citation format, read-scope, and "claims,
re-verified" framing are canonical in [run-store.md](run-store.md) § Reading prior runs.
Iteration-specific nuance: when a prior observation is load-bearing for THIS iteration's
verdict, a dispatched specialist re-verifies it and the coordinator merges the result as a
fresh local `OBS###` (not inline coordinator evidence work) so this run's citation chain
stays self-contained.

## Cross-iteration lead state

Default: carry every prior lead forward as a cited prior-iteration claim; change a lead's
state only when the new info or fresh evidence warrants it. Re-derive the verdict on the
merged evidence; never inherit the prior verdict. Owner-supplied telemetry — a raw log/record
paste from a human responder — that contradicts the prior verdict is itself a RE-DERIVATION
trigger: parse it into fresh local evidence and re-derive from its own authoritative source,
with the prior verdict entering only as an OPEN rival rung (`provenance=inherited`, per
grading-rubric.md Gate C anti-anchoring), never the frame the new evidence is scored against.

- `blocked-unreachable` → `open-answerable`: the new info is exactly what unblocks the
  lead (the human supplied the missing access, metric, or time window). This is the
  primary case iteration mode exists for — pursue it first.
- `closed-supported` stays settled unless the new info contradicts it; if it is the spine
  of the current verdict, re-check it cheaply via a dispatched probe.
- `closed-refuted` → reopened when the new info contradicts the earlier refutation.
- prior `open-answerable` (structural-cap-exhausted) leads: pick up if still material.

"Stale" means directly contradicted or mooted by the new info — not merely old. Supersede
only those. Do not discard still-valid prior corroboration (amnesia), and do not import a
prior `Confirmed`/`Likely-rooted` verdict as the answer (over-anchor).

## Capability map

Rebuild the CAPABILITY MAP for this iteration; access and capabilities can differ between
runs. Prior ACCESS STATUS is a claim, re-probed on the critical evidence path.

## Delta report

The `6_report` for an iteration is a delta/update bounded by the new Grader verdict: lead
with what changed since the last iteration and the updated verdict, including honest
downgrades (e.g. prior `Likely-rooted` → `Proximate-only`). When live posting is authorized, the incident post is likewise a delta and uses the posting capability's own idempotency/audit marker for that iteration; the Poster posts only to the provided incident record. See `subagents/poster.md` § Live incident-system posting.
